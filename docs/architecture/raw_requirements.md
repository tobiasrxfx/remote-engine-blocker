# REB — Requirements (Extracted)

## Functional Requirements (FR)

### FR-001 — State-driven remote engine immobilization with progressive blocking.
- **Type:** Functional  
- **Owner:** Group 4  
- **Functional Group:** State Machine and Control Logic  
- **Description:**  
  Upon receiving a valid lock command (manual, remote, or automatic theft detection), the REB system shall control engine immobilization according to the system state machine.  
  The state transitions shall occur as follows:
  1. **IDLE → THEFT_CONFIRMED**  
     Triggered when a theft condition is detected (manual activation, remote command via LTE/SMS/app, GPS mismatch, iButton alert, or sensor anomaly).
  2. **THEFT_CONFIRMED → BLOCKING**  
     The REB initiates the blocking procedure via CAN communication with the target ECU.  
     - If vehicle speed = 0 km/h: ignition/starter lock is requested.  
     - If vehicle speed > 0 km/h: gradual immobilization is requested via fuel pump ECU.
  3. **BLOCKING → BLOCKED**  
     The ECU confirms that the immobilization action has been executed and the vehicle is effectively prevented from operating.  
  During blocking, the REB shall send CAN commands through the TCU or directly to the ECU responsible for ignition or fuel control.
- **Rationale:**  
  To prevent vehicle theft by remotely or automatically immobilizing the vehicle using a controlled and safe state-based mechanism. The use of a state machine ensures predictable behavior and safe transitions during immobilization.
- **Pre-condition:** System state is IDLE and the vehicle systems are operating normally.
- **Stimulus / Input:** A valid engine lock request is received from one of the following sources:  
  - Remote command (4G/5G / SMS / mobile app)  
  - Local activation (vehicle panel)  
  - Automatic theft detection (GPS mismatch, iButton alert, sensor anomaly)
- **Response / Output:** The system shall:  
  1. Transition to THEFT_CONFIRMED.  
  2. Send CAN command to the blocking ECU via TCU or directly.  
  3. Execute immobilization depending on vehicle state:  
     - Ignition/starter disable when vehicle is stationary.  
     - Gradual fuel cut when vehicle is moving.  
  4. Record the event in system logs.  
  5. Send confirmation of immobilization status.
- **Post-condition:** The system state becomes BLOCKED and the engine cannot be restarted or continue normal operation.
- **Acceptance Criterion:**  
  MIL: SC0–SC3 simulation with automatic state transitions.  
  - SC0: System remains in IDLE under normal conditions.  
  - SC1: Theft event triggers THEFT_CONFIRMED transition.  
  - SC2: Blocking command triggers BLOCKING state and CAN request is sent.  
  - SC3: ECU confirms immobilization and system enters BLOCKED state.  
  Expected behavior: 100% of executions follow the defined state machine transitions.
- **Verification Method:** Inspection / Analysis-Simulation (MIL + SIL)
- **Priority:** MUST  
- **Risk:** High  
- **Difficulty:** Nominal  
- **Version:** v0.1  
- **Status:** Approved

### FR-002 — Validated theft confirmation prior to engine immobilization.
- **Type:** Functional  
- **Owner:** Group 4  
- **Functional Group:** State Machine and Control Logic  
- **Description:**  
  The REB system shall allow transition to the THEFT_CONFIRMED state only after receiving an explicit and valid theft confirmation signal.  
  The confirmation may originate from one of the following authorized sources:  
  - Manual activation from a local interface (vehicle panel or mobile application)  
  - Remote activation from backend services via LTE/4G/5G or SMS communication through the TCU  
  - Automatic activation triggered by predefined system rules based on sensor anomalies (e.g., glass break detection, abnormal acceleration, or intrusion detection)  
  Once a valid confirmation is received, the REB shall transition the system state from IDLE to THEFT_CONFIRMED.  
  No immobilization command shall be executed before the system enters the THEFT_CONFIRMED state.  
  After confirmation, the REB becomes authorized to initiate the immobilization procedure according to the blocking logic defined in the state machine.
- **Rationale:**  
  To prevent unintended engine immobilization and ensure that blocking actions are executed only after a verified theft event has been confirmed by an authorized source or trusted system rule.
- **Pre-condition:** The system is operating in IDLE state and no immobilization command has been executed.
- **Stimulus / Input:** A valid theft confirmation signal is received from one of the following sources:  
  - Authorized user command (local interface or mobile)  
  - Backend command received via TCU communication channel  
  - Automatic theft detection event generated by vehicle sensors
- **Response / Output:** The system shall:  
  1. Validate the authenticity of the theft confirmation request.  
  2. Transition the system state from IDLE to THEFT_CONFIRMED.  
  3. Record the confirmation event in the system log.  
  4. Enable the immobilization process that may subsequently transition the system to BLOCKING.
- **Post-condition:** The system state becomes THEFT_CONFIRMED, enabling the execution of immobilization procedures.
- **Acceptance Criterion:**  
  MIL: SC0–SC2 simulation.  
  - SC0: System remains in IDLE when no confirmation signal is present.  
  - SC1: Valid theft confirmation request triggers transition IDLE → THEFT_CONFIRMED.  
  - SC2: Invalid or unauthorized requests are rejected and the system remains in IDLE.  
  Expected behavior: All valid confirmations trigger the correct state transition and unauthorized triggers are rejected.
- **Verification Method:** Inspection / Analysis-Simulation (SIL)
- **Priority:** MUST  
- **Risk:** High  
- **Difficulty:** Nominal  
- **Version:** v0.1  
- **Status:** Approved

### FR-003 — Vehicle-condition-based state transition control for safe immobilization.
- **Type:** Functional  
- **Owner:** Group 4  
- **Functional Group:** State Machine and Control Logic  
- **Description:**  
  The REB system shall monitor vehicle operational signals to determine whether state transitions related to engine immobilization are permitted.  
  The following vehicle signals shall be used as decision inputs:  
  - Vehicle speed  
  - Ignition status  
  - Engine running status  
  These signals shall control the allowed transitions between system states.  
  The transition logic shall follow the rules below:
  1. **IDLE → THEFT_CONFIRMED**  
     Allowed after valid theft confirmation (as defined in RF-002).
  2. **THEFT_CONFIRMED → BLOCKING**  
     The REB shall determine the appropriate blocking strategy based on vehicle conditions:  
     - If vehicle speed = 0 km/h, the system shall initiate ignition/starter blocking via the ignition ECU.  
     - If vehicle speed > 0 km/h, the system shall initiate gradual immobilization via the fuel pump ECU.
  3. **BLOCKING → BLOCKED**  
     The transition shall occur after confirmation that the immobilization command has been executed by the ECU.  
  Transitions that could cause unsafe behavior shall be prevented. For example, direct ignition blocking shall not be allowed when the vehicle is moving.  
  All state transitions must be executed through CAN communication between the REB and the target ECU via the TCU or directly.
- **Rationale:**  
  To ensure safe engine immobilization by validating vehicle operating conditions before executing blocking actions, preventing unsafe engine shutdown during vehicle motion.
- **Pre-condition:** The system is operating in THEFT_CONFIRMED state and a blocking action is authorized.
- **Stimulus / Input:** Vehicle status signals received via CAN bus:  
  - Vehicle speed signal  
  - Ignition status signal  
  - Engine running signal
- **Response / Output:** The system shall:  
  - Evaluate the vehicle status signals.  
  - Determine whether the requested state transition is allowed.  
  - Select the appropriate blocking strategy: ignition/starter blocking for stationary vehicles; gradual fuel cut for moving vehicles.  
  - Send the corresponding immobilization command via CAN to the ECU.  
  - Record the decision and transition in the system log.
- **Post-condition:** The system transitions to BLOCKING, followed by BLOCKED once immobilization is confirmed.
- **Acceptance Criterion:**  
  MIL: SC0–SC3 simulation.  
  - SC0: Vehicle moving → ignition blocking request is rejected.  
  - SC1: Vehicle stationary → ignition blocking allowed.  
  - SC2: Vehicle moving → gradual immobilization initiated.  
  - SC3: ECU confirms immobilization → system transitions to BLOCKED.  
  Expected behavior: state transitions occur only when vehicle conditions satisfy safety rules.
- **Verification Method:** Analysis-Simulation (SIL)
- **Priority:** MUST  
- **Risk:** High  
- **Difficulty:** Nominal  
- **Version:** v0.1  
- **Status:** Approved

### FR-005 — Remote Activation via 4G/5G (TCU/Back-end).
- **Type:** Functional  
- **Owner:** Group 4  
- **Functional Group:** Activation and Detection  
- **Description:**  
  The system shall support remote activation of the REB via cellular channels (4G/5G) through the Telematics Control Unit (TCU). The REB must process the activation trigger received from the TCU, send an acknowledgment (ACK) signal, and provide the real-time execution status for back-end monitoring.
- **Rationale:** Critical for anti-theft operations, allowing the vehicle to be secured remotely even when the owner is not physically present.
- **Pre-condition:** System in IDLE state; TCU connected to the cellular network and communicating with the vehicle's internal bus.
- **Stimulus / Input:** Activation command bitstream received from the TCU via CAN/Ethernet bus.
- **Response / Output:** Transition to THEFT_CONFIRMED state; Transmission of Acknowledgment (ACK) and Status signals back to the TCU.
- **Post-condition:** System proceeds to the transition sequence; Event is stored in the event_log.slx via NVM.
- **Acceptance Criterion:**  
  MIL/SIL: Simulate the reception of the specific activation CAN message from the TCU. Verify that the state transitions to THEFT_CONFIRMED within 100ms and the ACK signal is triggered.
- **Verification Method:** Analysis-Simulation (MIL)
- **Priority:** MUST  
- **Risk:** High  
- **Difficulty:** Nominal  
- **Version:** v0.1

### FR-007 — Automatic Activation via Sensor
- **Type:** Functional  
- **Owner:** Group 4  
- **Functional Group:** Activation and Detection  
- **Description:**  
  The system shall support automatic activation of the REB based on vehicular sensor events, specifically door/window breach signals from the Body Control Module (BCM) and impact/tilt data from the accelerometer. The logic must implement configurable rules and hysteresis to filter out false positives (e.g., minor vibrations).
- **Rationale:** Ensures the vehicle is protected even if the owner is unable to send a remote command, detecting physical intrusion or unauthorized movement.
- **Pre-condition:** System in IDLE state; BCM and Accelerometer signals available on the CAN bus.

- **Response / Output:** Transition to THEFT_CONFIRMED state; Activation of internal notification timers.
- **Post-condition:** Trigger event recorded in the event_log.slx via NVM; Notification signal broadcasted to the instrument cluster.
- **Acceptance Criterion:**

- **Verification Method:** Analysis-Simulation (MIL)

### FR-008 — Pre-notification and Reversion Window (Auto-trigger)
- **Type:** Functional  
- **Owner:** Group 4  
- **Functional Group:** Actuation and Safety  
- **Description:**  
  Upon an automatic trigger (FR-007), the system shall issue a prior notification and wait for a 60-second tolerance window before transitioning to the BLOCKING state. During this period, the process can be reversed and the system returned to IDLE via a security password on the infotainment system or an authenticated mobile app command.
- **Rationale:** Prevents unintended vehicle immobilization in case of sensor false positives, allowing the legitimate user to safely abort the process before any engine intervention.
- **Pre-condition:** System in THEFT_CONFIRMED state triggered specifically by automatic sensor rules.
- **Stimulus / Input:** Timer count reaching 60s (to proceed) OR valid PIN/Abort Command received (to cancel).
- **Response / Output:** Transition to BLOCKING (if time expires) OR return to IDLE (if aborted).
- **Post-condition:** Reversion event or transition to blocking phase logged in the event_log.slx.
- **Acceptance Criterion:**  
  MIL: Trigger an automatic event and verify the 60s countdown initiation. Send an abort command at 30s and confirm the state returns to IDLE with no torque reduction.
- **Verification Method:** Analysis-Simulation (MIL)
- **Priority:** CAN  
- **Risk:** Medium  
- **Difficulty:** Nominal  
- **Version:** v0.1  
- **Status:** Approved

### FR-009 — Minimum fuel safety floor guarantee during in-motion gradual blocking (torque derating).
- **Type:** Functional  
- **Owner:** Group 4  
- **Functional Group:** Actuation and Safety  
- **Description:**  
  The Actuator_Interface shall enforce a minimum fuel safety floor during the BLOCKING state when the vehicle is in motion, by guaranteeing derate_pct_out >= 10% while vehicle_speed > v_stop_kmh = 0.5 km/h and blocking_state = BLOCKING. The enforcement shall be implemented as derate_pct_out = max(derate_pct_in, min_derate) in the Fuel_Control subblock, where min_derate = 10%. This requirement supports the gradual in-motion blocking behavior by ensuring controlled deceleration without engine shutdown and preserving steering and braking assist, and shall remain active regardless of any upstream controller output while the above conditions hold.
- **Rationale:**  
  A minimum fuel floor during in-motion gradual blocking prevents complete engine stall while the vehicle is moving, which could lead to loss of steering and braking assist. By enforcing the constraint in Actuator_Interface as an independent layer, the safety floor remains effective even in the presence of upstream logic faults, while still allowing gradual deceleration via fuel/torque derating.
- **Pre-condition:**  
  - vehicle_speed > v_stop_kmh = 0.5 km/h  
  - blocking_state = BLOCKING  
  - derating_active = true
- **Stimulus / Input:**  
  - derate_pct_in from Fuel_Control  
  - vehicle_speed  
  - blocking_state
- **Response / Output:** derate_pct_out = max(derate_pct_in, 10%) while vehicle_speed > 0.5 km/h and blocking_state = BLOCKING.
- **Post-condition:** derate_pct_log >= 10% for all samples with vehicle_speed > 0.5 km/h while blocking_state = BLOCKING.
- **Acceptance Criterion:**  
  MIL: Set blocking_state = BLOCKING. Inject derate_pct_in = 0% with vehicle_speed = 5 km/h.  
  Criteria:  
  (a) derate_pct_out = 10%;  
  (b) 0 samples with derate_pct_out below 10% while vehicle_speed > 0.5 km/h and blocking_state = BLOCKING across all SC01–SC08 scenarios and 500 Monte Carlo runs.
- **Verification Method:** Analysis-Simulation (MIL)
- **Priority:** MUST  
- **Risk:** High — failure causes loss of vehicle control  
- **Difficulty:** Nominal  
- **Version:** v0.1  
- **Status:** Approved

### FR-010 — Transition from in-motion gradual blocking to parked restart inhibit upon safe stop.
- **Type:** Functional  
- **Owner:** Group 4  
- **Functional Group:** Actuation and Safety  
- **Description:**  
  The system shall, while in BLOCKING, detect when the vehicle reaches a safe stop condition and transition to the parked blocking logic to prevent any engine restart. A safe stop condition shall be confirmed when vehicle_speed <= v_stop_kmh = 0.5 km/h continuously for t_stop_confirm_s = 120 s. Immediately upon safe stop confirmation (t_transition_max_ms = 0 ms), the system shall (a) assert parked_block_request = TRUE and (b) issue the configured CAN command path to enable starter restart inhibit. The parked blocking logic shall remain active until an authorized release/unblock command is received.
- **Rationale:** After a controlled in-motion derating, the system must prevent the vehicle from being restarted once it has stopped, ensuring the theft mitigation objective while avoiding unsafe actions during motion.
- **Pre-condition:**  
  - blocking_state = BLOCKING  
  - vehicle_speed <= v_stop_kmh = 0.5 km/h continuously for t_stop_confirm_s = 120 s
- **Stimulus / Input:**  
  - vehicle_speed  
  - blocking_state  
  - start_request (driver attempt to start)
- **Response / Output:**  
  - parked_block_request = TRUE immediately after safe stop confirmation  
  - CAN command issued to enable restart inhibit (starter inhibit)  
  - restart_inhibit_active = TRUE while parked blocking logic is active  
  - start_permitted = FALSE when restart_inhibit_active = TRUE
- **Post-condition:**  
  - restart_inhibit_active remains TRUE after safe stop while blocking remains armed  
  - any engine start/restart request is inhibited (start_permitted = FALSE) while restart_inhibit_active = TRUE
- **Acceptance Criterion:**  
  MIL: Set blocking_state = BLOCKING. Drive vehicle_speed from 10 km/h to 0 km/h and hold vehicle_speed = 0 km/h for 121 s.  
  Criteria:  
  (a) parked_block_request transitions to TRUE in the first simulation sample immediately after the 120 s safe stop confirmation is satisfied;  
  (b) restart_inhibit_active = TRUE for all samples after parked_block_request = TRUE;  
  (c) inject start_request = TRUE after restart_inhibit_active = TRUE and verify start_permitted = FALSE for all samples while restart_inhibit_active = TRUE;  
  (d) 0 samples with start_permitted = TRUE while restart_inhibit_active = TRUE across all SC01–SC08 scenarios and 500 Monte Carlo runs.
- **Verification Method:** Analysis-Simulation (MIL)
- **Priority:** MUST  
- **Risk:** High  
- **Difficulty:** Nominal  
- **Version:** v0.1  
- **Status:** Approved

### FR-011 — Definitive transition to BLOCKED after 120 s safe-stop dwell with direct starter inhibit and release of fuel derating.
- **Type:** Functional  
- **Owner:** Group 4  
- **Functional Group:** Actuation and Safety  
- **Description:**  
  The system shall, while in BLOCKING, enforce gradual immobilization via fuel control to prevent vehicle movement and, upon a confirmed safe-stop dwell, transition to BLOCKED and apply a definitive starter inhibit command directly to the starter ECU. A safe stop dwell shall be defined as vehicle_speed <= v_stop_kmh = 0.5 km/h continuously for t_blocked_dwell_s = 120 s. The dwell timer shall start when vehicle_speed first satisfies the threshold while blocking_state = BLOCKING, and shall reset to 0 if vehicle_speed > v_stop_kmh at any time before completion.  
  During the dwell interval (i.e., before t_blocked_dwell_s is satisfied), the system shall continue to apply fuel derating via CAN to the fuel pump ECU (or equivalent) as the active immobilization mechanism, such that vehicle propulsion is effectively prevented while maintaining engine operation (no intentional engine shutdown).  
  Immediately upon dwell completion (t_transition_max_ms = 0 ms), the system shall transition to BLOCKED, issue a direct CAN command to the starter ECU to enable starter inhibit (restart prevention), and shall stop issuing fuel derating commands (fuel derating shall be released/disabled) once BLOCKED is entered. The starter inhibit shall remain active until an authorized release/unblock command is received.
- **Rationale:**  
  Maintaining fuel derating during BLOCKING prevents the vehicle from regaining propulsion before definitive immobilization is permitted. Transitioning to BLOCKED after a stable safe-stop dwell enables a robust theft-mitigation state by preventing any restart via starter inhibit, while eliminating unnecessary fuel derating actuation once definitive immobilization is achieved.
- **Pre-condition:**  
  - blocking_state = BLOCKING  
  - vehicle_speed and v_stop_kmh available and valid  
  - CAN communication available to fuel pump ECU and starter ECU  
  - driver_torque_request available (for MIL attempts to move)
- **Stimulus / Input:**  
  - vehicle_speed  
  - blocking_state  
  - driver_torque_request  
  - derate_pct_in (from upstream Fuel_Control logic)  
  - unblock_request (authorized release command)
- **Response / Output:**  
  - While blocking_state = BLOCKING and vehicle_speed <= 0.5 km/h and dwell_timer < 120 s:  
    - fuel_derating_active = TRUE  
    - derate_pct_out applied via CAN to fuel pump ECU  
  - If vehicle_speed > 0.5 km/h at any time before dwell_timer reaches 120 s:  
    - dwell_timer resets to 0 s  
  - When dwell_timer reaches 120 s (continuous):  
    - blocking_state transitions to BLOCKED immediately  
    - starter_inhibit_active = TRUE and starter_inhibit_cmd issued via direct CAN to starter ECU  
    - fuel_derating_active = FALSE and fuel derating commands are no longer issued
- **Post-condition:**  
  - blocking_state = BLOCKED after dwell completion  
  - starter_inhibit_active remains TRUE until authorized unblock_request is received  
  - fuel_derating_active = FALSE for all samples with blocking_state = BLOCKED
- **Acceptance Criterion:**  
  MIL: Set blocking_state = BLOCKING. Apply a drive-down profile to vehicle_speed from 10 km/h to 0 km/h, then hold vehicle_speed = 0 km/h (<= 0.5 km/h) continuously for 121 s.  
  Criteria:  
  (a) dwell_timer reaches 120 s without reset while vehicle_speed <= 0.5 km/h;  
  (b) blocking_state transitions to BLOCKED in the first simulation sample immediately after dwell completion;  
  (c) starter_inhibit_active = TRUE and starter_inhibit_cmd asserted immediately upon entering BLOCKED;  
  (d) fuel_derating_active = TRUE for all samples in BLOCKING prior to BLOCKED, and fuel_derating_active = FALSE for all samples in BLOCKED;  
  (e) inject driver_torque_request > 0 after entering BLOCKED and verify restart remains inhibited (starter_inhibit_active = TRUE) and no fuel derating commands are issued;  
  (f) 0 samples with starter_inhibit_active = FALSE while blocking_state = BLOCKED across all SC01–SC08 scenarios and 500 Monte Carlo runs.
- **Verification Method:** Analysis-Simulation (MIL)
- **Priority:** MUST  
- **Risk:** High  
- **Difficulty:** Difficult  
- **Version:** v0.1  
- **Status:** Approved

### FR-012 — 90 s warning and reversal window prior to entering BLOCKING after an automatic blocking trigger.
- **Type:** Functional  
- **Owner:** Group 4  
- **Functional Group:** Actuation and Safety  
- **Description:**  
  The system shall, when activated by an automatic blocking trigger, issue a pre-blocking warning notification and enforce a tolerance window of t_reversal_window_s = 90 s before transitioning from THEFT_CONFIRMED to BLOCKING. Immediately upon the automatic trigger placing the system in THEFT_CONFIRMED, the system shall (a) present an alert on the Infotainment interface indicating imminent blocking and (b) start a reversal timer.  
  During the reversal window, the system shall allow the driver to interrupt and reverse the blocking process by entering a valid security password on the Infotainment interface. If the security password is validated within the reversal window and before any blocking actuation command is issued, the system shall cancel the pending transition to BLOCKING, clear the automatic-trigger event, and transition to IDLE.  
  If the reversal window expires with no valid password entry, the system shall transition to BLOCKING. If a password is entered after the 90 s window, the system shall permit abort/reversal only if the system has not yet issued any blocking actuation command; otherwise, the password entry shall be rejected for reversal purposes and the system shall continue its blocking sequence.
- **Rationale:**  
  A pre-blocking warning with a reversal window reduces the likelihood of unintended immobilization caused by false-positive automatic triggers, while still enabling timely theft mitigation. Conditioning reversal on “no blocking actuation issued” prevents unsafe or inconsistent partial-blocking scenarios.
- **Pre-condition:**  
  - blocking_state transitions to THEFT_CONFIRMED due to an automatic blocking trigger  
  - Infotainment interface available for alert display and password entry  
  - blocking_actuation_issued = FALSE at the start of the reversal window
- **Stimulus / Input:**  
  - auto_block_trigger event  
  - blocking_state  
  - infotainment_password_entry  
  - blocking_actuation_issued
- **Response / Output:**  
  - Upon auto_block_trigger:  
    - blocking_state = THEFT_CONFIRMED  
    - pre_block_alert_active = TRUE (Infotainment warning displayed)  
    - reversal_timer starts (0 → 90 s)  
  - While reversal_timer < 90 s and blocking_actuation_issued = FALSE:  
    - if password_valid = TRUE then:  
      - cancel_pending_block = TRUE  
      - blocking_state transitions to IDLE  
      - pre_block_alert_active = FALSE  
  - When reversal_timer reaches 90 s and no valid password was accepted:  
    - blocking_state transitions to BLOCKING  
  - If password_valid = TRUE after 90 s:  
    - if blocking_actuation_issued = FALSE then allow reversal to IDLE  
    - else reject reversal and continue blocking sequence
- **Post-condition:**  
  - If password accepted for reversal: blocking_state = IDLE and no blocking actuation commands have been issued  
  - If reversal not performed: blocking_state = BLOCKING after 90 s and pre-block alert is cleared/updated per HMI policy
- **Acceptance Criterion:**  
  MIL: Trigger auto_block_trigger and confirm the system enters THEFT_CONFIRMED at t = 0 s with pre_block_alert_active = TRUE.  
  Criteria:  
  (a) The system remains in THEFT_CONFIRMED for exactly 90 s (±1 simulation sample) when no password is entered; at t = 90 s it transitions to BLOCKING.  
  (b) Inject a valid infotainment_password_entry at t = 30 s with blocking_actuation_issued = FALSE. Verify the system transitions to IDLE within 1 simulation sample, cancel_pending_block = TRUE, and 0 blocking actuation commands are issued.  
  (c) Inject a valid infotainment_password_entry at t = 95 s:  
      - Case C1: blocking_actuation_issued = FALSE → verify reversal to IDLE is allowed.  
      - Case C2: blocking_actuation_issued = TRUE → verify reversal is rejected (state does not return to IDLE) and blocking continues.  
  (d) Across all SC01–SC08 scenarios and 500 Monte Carlo runs, verify 0 samples where blocking_state transitions to BLOCKING before t = 90 s without an explicit non-reversed condition.
- **Verification Method:** Analysis-Simulation (MIL)
- **Priority:** SHOULD  
- **Risk:** Medium  
- **Difficulty:** Nominal  
- **Version:** v0.1  
- **Status:** Approved

### FR-013 — Visual and audible alerts for imminent blocking notification.
- **Type:** Functional  
- **Owner:** Group 4  
- **Functional Group:** Actuation and Safety  
- **Description:**  
  After the minimum safety time (defined in FR-012) has elapsed, the system must activate multimodal alerts to inform the driver of the imminent locking. The system must: (a) display a critical alert message on the infotainment display, (b) activate the horn in flashing mode, and (c) activate the hazard warning lights. These alerts must remain active while the system is in the transition state to effective locking, unless a valid reversal is detected.
- **Rationale:**  
  Ensure that the driver has a clear perception (visual and auditory) that the vehicle will be immobilized, allowing them to take safety measures (pull the vehicle over) or enter the reverse code to avoid stopping in an unsafe location.
- **Pre-condition:**  
  - The safety timer t_min_safety (FR-012) has expired.  
  - The blocking_state transitions to the pre-lock state (e.g., BLOCK_ALERT).  
  - The system did not receive a rollback command/valid password.
- **Stimulus / Input:**  
  - safety_timer_expired == TRUE  
  - blocking_state  
  - reversal_password_status
- **Response / Output:**  
  - When safety_timer_expired is detected:  
    - hmi_display_alert = "ALERT: IMMINENT BLOCKAGE"  
    - horn_command = INTERMITTENT (frequency of 1 Hz)  
    - hazard_lights_command = ON  
  - If password_valid == TRUE during the alert:  
    - horn_command = OFF  
    - hazard_lights_command = OFF  
    - hmi_display_alert = CLEAR
- **Post-condition:** Alerts remain active until the blocking_state transitions to BLOCKED or returns to IDLE via rollback.
- **Acceptance Criterion:**  
  MIL: Simulate the expiration of the safety timer and verify the output signals.  
  Criteria:  
  (a) At the exact instant that safety_timer_expired becomes TRUE, validate that hazard_lights_command and horn_command are activated within 100 ms.  
  (b) Verify that the text message in the Infotainment system exactly matches the HMI data dictionary.  
  (c) Inject password_valid = TRUE and confirm that all alert outputs (visual and audible) cease in less than 1 simulation cycle.
- **Verification Method:** Analysis-Simulation (MIL)
- **Priority:** CAN  
- **Risk:** Low  
- **Difficulty:** Nominal  
- **Version:** v0.1  
- **Status:** Approved

## Non-Functional Requirements (NFR)

### NFR-SEC-001 — Anti-replay and integrity protection for remote commands
- **Type:** Non-Functional  
- **Owner:** Group 4  
- **Description:**  
  The system shall implement anti-replay mechanisms (e.g., nonces, monotonic counters, or message expiration) and cryptographic integrity validation (e.g., HMAC or Digital Signatures) for all control commands received via cellular data and SMS.
- **Rationale:**  
  To prevent unauthorized actors from intercepting and re-transmitting legitimate commands (replay attacks) or injecting forged messages, ensuring that only authentic, unique, and unaltered instructions from authorized sources are executed.
- **Metric:** Percentage of replayed or forged packets rejected by the system = 100%
- **Acceptance Criterion:** Security protocol analysis and penetration testing — the system must successfully reject 100% of replayed messages and altered payloads during simulated attacks.
- **Verification Method:** Analysis-Simulation / Demonstration
- **Priority:** SHOULD  
- **Risk:** High — lack of anti-replay mechanisms allows unauthorized vehicle immobilization or hijacking  
- **Difficulty:** Difficult  
- **Version:** v0.1  
- **Status:** Approved

### NFR-SAF-001 — Prohibition of abrupt engine blocking in motion
- **Type:** Non-Functional  
- **Owner:** Group 4  
- **Description:**  
  The system shall not execute a starter motor block while the vehicle is in motion (v > 0); only gradual blocking (progressive power/torque reduction) is permitted while the vehicle is moving.
- **Rationale:**  
  Sudden immobilization at high speeds poses a severe risk of collisions and loss of vehicle control, endangering passengers and other road users.
- **Metric:** Velocity at starter-cut signal activation = 0 km/h
- **Acceptance Criterion:** HIL/MIL Simulation: Ensure that the starter-cut actuation signal remains FALSE across all scenarios where vehicle_speed > 0.
- **Verification Method:** Analysis-Simulation
- **Priority:** MUST  
- **Risk:** High — abrupt blocking in motion is a safety-critical failure  
- **Difficulty:** Nominal  
- **Version:** v0.1  
- **Status:** Approved

### NFR-SAF-002 — Fail-safe operation under signal loss
- **Type:** Non-Functional  
- **Owner:** Group 4  
- **Description:**  
  In the absence or corruption of critical safety signals (e.g., vehicle speed unavailable via CAN), the REB shall enter a conservative mode and inhibit any blocking actions that could induce physical risk.
- **Rationale:**  
  Decisions made on faulty or missing data can lead to unintended and unsafe vehicle behavior.
- **Metric:** Block inhibition = TRUE when signal_status != VALID
- **Acceptance Criterion:** Inject signal faults (timeout/invalid data) during simulation and verify that the system transitions to a safe state, preventing any new blocking commands.
- **Verification Method:** Analysis-Simulation (SIL)
- **Priority:** MUST  
- **Risk:** High — failure to handle signal loss may lead to accidental blocking  
- **Difficulty:** Nominal  
- **Version:** v0.1  
- **Status:** Approved

### NFR-REL-001 — Post-Reset State Recovery and Persistence
- **Type:** Non-Functional  
- **Owner:** Group 4  
- **Description:**  
  The Remote Engine Blocker (REB) system shall persistently store its current operational state and critical context parameters in Non-Volatile Memory (NVM) upon every state transition. Following any reset, watchdog trigger, or abrupt power loss (e.g., battery disconnection), the system shall deterministically initialize, read the NVM, and resume operation from the last valid persisted state.
- **Rationale:**  
  To prevent malicious actors from bypassing the immobilization by intentionally disconnecting and reconnecting the vehicle's battery or forcing an ECU hardware reset.
- **Metric:** 100% of state transitions are successfully written to persistent storage before executing the next logical instruction. Software recovers the last state upon initialization.
- **Acceptance Criterion:** If power is abruptly cut while the REB is in the "BLOCKED" state, upon power restoration, the REB must automatically re-enter the "BLOCKED" state and resume transmitting the immobilization CAN frames without any user interaction or re-authentication.
- **Verification Method:** Analysis-Simulation
- **Priority:** MUST  
- **Risk:** High — failure to recover the locked state after a power drop could allow bypass via ECU reset  
- **Difficulty:** Nominal  
- **Version:** v0.1  
- **Status:** Approved

### NFR-INFO-001 — Secure Diagnostic Logging and Event Tracing
- **Type:** Non-Functional  
- **Owner:** Group 4  
- **Description:**  
  The REB system shall generate and store diagnostic logs and event records (e.g., rejected blocking commands due to high speed and execution failures) in persistent memory. These logs must provide sufficient context for field diagnostics and be retrievable via diagnostic services. The system must strictly ensure that no cryptographic secrets, user credentials, or plaintext sensitive data are exposed in the logs.
- **Rationale:**  
  To enable post-incident forensic analysis and facilitate troubleshooting by authorized technicians in the field, without compromising the cybersecurity of the vehicle or the user's privacy.
- **Metric:** 100% of critical state transitions and rejected blocking attempts are logged. 0% of sensitive authentication data is stored in plaintext.
- **Acceptance Criterion:** When a blocking command is rejected (e.g., because the vehicle speed is 50 km/h), the system must log a specific event detailing the timestamp and the exact reason for rejection. An authorized diagnostic tool must be able to read this log without extracting any cryptographic keys.
- **Verification Method:** Analysis-Simulation
- **Priority:** MUST  
- **Risk:** Medium — failure hinders troubleshooting/forensics  
- **Difficulty:** Nominal  
- **Version:** v0.1  
- **Status:** Approved
